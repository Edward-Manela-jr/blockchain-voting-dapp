## 1. Problem Definition & Real-World Application

**Problem Statement:**
Traditional voting systems rely on centralized authorities (governments, school boards, organizations) to manage, count, and verify votes. These systems are prone to:
- **Trust issues**: Administrators have unchecked power to alter results.
- **Inefficiency**: Manual counting is slow and error-prone.
- **Transparency gaps**: Voters cannot independently verify that their vote was counted correctly.
- **Single point of failure**: A compromised central server means the entire election is compromised.

**Solution:**
A Decentralized Voting DApp built on the Ethereum blockchain that addresses each of these problems:
- **Immutability**: Once a vote is cast, it is permanently recorded on-chain. No one — including the admin — can alter it.
- **Transparency**: Every vote is a verifiable blockchain transaction on a public ledger.
- **Trustless Verification**: Smart contract logic enforces election rules automatically (one vote per person, admin restrictions, candidate locking) without relying on human oversight.
- **Decentralized execution**: The contract runs on the Ethereum Virtual Machine across all network nodes, eliminating any single point of failure.

## 2. Architecture Overview

### System Components

The platform consists of three layers:

```
[User Browser + MetaMask Wallet]
        |
        | (ethers.js / JSON-RPC)
        v
[React Frontend - voting-dapp]
        |
        | (Contract ABI + Address)
        v
[Solidity Smart Contract - Voting.sol]
        |
        | (Deployed on)
        v
[Local Ethereum Network - Hardhat Node]
```

**Layer 1 — Smart Contract (Voting.sol):**
The core business logic lives entirely on-chain. The contract manages candidates, voter registration, vote casting, and result retrieval. Once deployed, the contract's rules cannot be changed.

**Layer 2 — React Frontend (voting-dapp/src/App.js):**
A single-page application that provides role-based interfaces for admins and voters. Communicates with the contract through ethers.js, which translates JavaScript calls into Ethereum transactions.

**Layer 3 — Wallet & Authentication (MetaMask):**
MetaMask acts as the identity layer. There are no usernames or passwords — the user's Ethereum address is their identity, and their private key signs every transaction. This is the cryptographic foundation of the entire system.

### Data Flow

1. **Admin adds candidates** → MetaMask signs the transaction → Hardhat node validates the `onlyAdmin` modifier → Candidate stored in `candidates` mapping
2. **Admin registers voters** → Voter address written to `registeredVoters` mapping on-chain
3. **Voter casts a vote** → Contract checks: election active, voter registered, not already voted, not admin, valid candidate ID → Vote count incremented → `votedEvent` emitted
4. **Results queried** → `getVotes()` reverts if election is still active; returns counts after election ends

## 3. Smart Contract Design

### Contract: Voting.sol (Solidity ^0.8.27)

**State Variables:**

| Variable | Type | Purpose |
|---|---|---|
| `admin` | `address` | The deployer's address, set in the constructor. Only this address can manage the election. |
| `candidates` | `mapping(uint => Candidate)` | Maps candidate IDs (starting at 1) to a struct containing `id`, `name`, and `voteCount`. |
| `candidatesCount` | `uint` | Tracks the total number of candidates. Also serves as the ID counter. |
| `registeredVoters` | `mapping(address => bool)` | Tracks which addresses are eligible to vote. |
| `voters` | `mapping(address => bool)` | Tracks which addresses have already cast a vote. |
| `hasVoted` | `mapping(address => bool)` | Secondary flag for vote status (used by frontend). |
| `electionActive` | `bool` | Controls the election lifecycle — `false` by default, toggled by admin. |

**Key Functions:**

| Function | Access | Description |
|---|---|---|
| `addCandidate(string _name)` | Admin only | Adds a candidate. Rejects duplicates (string comparison via `keccak256`). Blocked once election starts. |
| `registerVoter(address _voter)` | Admin only | Marks an address as eligible to vote. |
| `startElection()` | Admin only | Activates voting. Requires at least one candidate. |
| `endElection()` | Admin only | Deactivates voting. Results become queryable. |
| `vote(uint _candidateId)` | Registered voters | Casts a vote. Enforces: election active, voter registered, not already voted, not admin, valid candidate ID. Emits `votedEvent`. |
| `getVotes(uint _candidateId)` | Public (after election) | Returns vote count for a candidate. Reverts while election is active to prevent influencing voters. |

**Access Control:**
The `onlyAdmin` modifier gates all management functions. It uses `require(msg.sender == admin)` to verify the caller is the deployer. This is a gas-efficient alternative to OpenZeppelin's `Ownable` pattern, suitable for a single-admin system.

**Election Lifecycle State Machine:**

```
[Not Started] --startElection()--> [Active] --endElection()--> [Ended]
     |                                |                           |
  addCandidate: YES             addCandidate: NO           addCandidate: NO
  registerVoter: YES            registerVoter: YES         registerVoter: YES
  vote: NO                      vote: YES                  vote: NO
  getVotes: returns 0           getVotes: REVERTS          getVotes: returns count
```

## 4. Security Measures

### On-Chain Security

1. **Access Control via `onlyAdmin` Modifier:**
   All administrative functions (adding candidates, registering voters, starting/ending the election) are gated by the `onlyAdmin` modifier. This ensures that only the deployer wallet can manage the election. Any other address calling these functions triggers a `require` revert.

2. **Double-Vote Prevention:**
   The `voters[msg.sender]` mapping is checked before every vote and set to `true` after a successful vote. This is an atomic check-and-set within a single transaction, making double-voting impossible even if a user submits multiple transactions rapidly.

3. **Admin Vote Exclusion:**
   The contract explicitly prevents the admin from voting (`require(msg.sender != admin)`). This separation of powers ensures the election administrator cannot influence the outcome.

4. **Candidate Locking:**
   Once `startElection()` is called, `addCandidate()` reverts. This prevents the admin from adding candidates mid-election, which could manipulate vote distribution.

5. **Result Concealment During Voting:**
   `getVotes()` reverts while `electionActive` is true. This prevents partial results from influencing voter behaviour — a well-known concern in election design.

6. **Candidate ID Validation:**
   The `vote()` function checks `_candidateId > 0 && _candidateId <= candidatesCount`, preventing votes for non-existent candidates.

7. **Duplicate Candidate Prevention:**
   `addCandidate()` iterates existing candidates and compares names using `keccak256(bytes(...))` hashing. This prevents the admin from adding the same candidate twice to split votes.

8. **Immutable Vote Record:**
   Once a vote increments `candidates[_candidateId].voteCount`, it is permanently stored on-chain. Neither the admin nor any external party can modify this value after the fact.

### Off-Chain Security

9. **Nonce Management:**
    The `sendTx` helper in `contract.js` queries the nonce directly from the Hardhat node via `JsonRpcProvider`, bypassing MetaMask's cached nonce. This prevents transaction failures caused by stale nonce values after chain restarts.

10. **Fresh Provider per Interaction:**
    `getContract()` creates a new `BrowserProvider` instance on every call rather than caching a stale provider. This ensures the signer always reflects the currently connected MetaMask account.

11. **Wallet Change Detection:**
    The frontend subscribes to MetaMask's `accountsChanged` event and re-fetches all contract state (admin status, registration, vote status) when the user switches wallets. This prevents stale UI state from causing confusion or errors.

## 5. Blockchain Tools Analysis

### Hardhat vs. Ganache
We chose **Hardhat** over Ganache for the development environment:
- **Built-in `console.log`** in Solidity for contract debugging (not available in Ganache)
- **Faster block mining** with instant mining mode for rapid development cycles
- **Integrated testing framework** with fixtures and snapshot/revert for isolated test cases
- **Deterministic accounts** with the same addresses and private keys on every restart
- **Better error messages** with human-readable revert reason strings

### Ethers.js v6 vs. Web3.js
We chose **ethers.js v6** over Web3.js:
- **Separation of concerns**: ethers.js distinguishes between Provider (read) and Signer (write), making the code more explicit about which operations require a wallet
- **Smaller bundle size**: ethers.js is significantly lighter than Web3.js
- **Better TypeScript support** and more consistent API design
- **Native BigInt support** in v6 eliminates the need for BigNumber wrapper objects

### MetaMask as Identity Layer
MetaMask serves as the authentication and authorization layer:
- **No password database**: The user's Ethereum address is their identity; their private key proves ownership
- **Transaction signing**: Every state-changing action requires explicit user approval via the MetaMask popup
- **Account switching**: The `accountsChanged` event enables seamless role switching without page reloads

### React + Tailwind CSS
- **React**: Component-based architecture with hooks (`useState`, `useEffect`, `useCallback`) for managing contract state and UI reactivity
- **Tailwind CSS**: Utility-first CSS framework that enabled a professional dark-themed UI without writing custom stylesheets. Gradient backgrounds, rounded cards, and responsive grid layouts are composed directly in JSX.

## 7. Gas Optimisation

### Techniques Applied

1. **Mappings over Arrays for Lookups:**
   Used `mapping(address => bool)` for `registeredVoters`, `voters`, and `hasVoted`. Mapping lookups are O(1) constant-time operations, whereas array iteration would be O(n) — a significant gas saving as the voter count grows.

2. **Memory over Storage for Temporary Data:**
   Function parameters like `string memory _name` use memory allocation rather than storage. Memory is cleared after the function call and costs significantly less gas than writing to persistent storage.

3. **Early Revert with Require Statements:**
   `require` checks are ordered to fail as early as possible in each function. For example, in `vote()`, the admin check comes first, followed by election state, then registration, then double-vote check. This ensures that invalid transactions consume minimal gas before reverting.

4. **Minimal Storage Writes:**
   The `vote()` function performs exactly two storage writes (`voters[msg.sender] = true` and `candidates[_candidateId].voteCount++`) and one event emission. No redundant state changes.

5. **Compiler Optimiser:**
   The Hardhat configuration enables the Solidity optimiser with 200 runs, which reduces bytecode size and gas costs for frequently-called functions.

## 9. Personal Reflection & Challenges

### Challenges Encountered

**Hardhat Environment Setup:**
- **Hardhat v3 vs v2 Dependency Conflict**: The initial `npm init` installed Hardhat v3, which introduced ESM modules, a new plugin system, and different configuration formats. However, nearly all tutorials, university resources, and community examples target Hardhat v2. This caused immediate incompatibilities — plugins would install but fail at runtime. We resolved this by downgrading to Hardhat 2.22 (`npm install hardhat@2.22.0 --legacy-peer-deps`), aligning our environment with the stable ecosystem.
- **ESM vs CommonJS Module Error**: After the initial setup, we encountered `require is not defined in ES module scope`. Hardhat v3 had silently switched the project to ESM mode, but our config and test files used CommonJS `require()` syntax. We fixed this by ensuring `hardhat.config.js` and all scripts used CommonJS, consistent with Hardhat v2 expectations.
- **Plugin Dependency Chain Explosion**: Installing `@nomicfoundation/hardhat-toolbox` pulled in a deep tree of sub-dependencies (hardhat-ethers, chai, ts-node, typechain, solidity-coverage, etc.). Several of these had version conflicts with each other, leading to cryptic errors. The fix was a clean rebuild: delete `node_modules` and `package-lock.json`, then `npm install` from scratch.
- **Foundry Template Contamination**: The Hardhat v3 initialisation scaffolded a `Counter.t.sol` test file that uses Foundry's `forge-std` library — a completely different toolchain. Running `npx hardhat test` would fail because Hardhat cannot resolve Foundry imports. We removed the Foundry-specific files and replaced them with Hardhat-native tests.
- **Hardhat Cache Conflicts**: After modifying contracts, old compiled artifacts sometimes persisted in the `cache/` and `artifacts/` directories, causing "contract not found" or ABI mismatch errors. Running `npx hardhat clean` before recompiling resolved these issues.
- **Node.js Version Mismatch**: Hardhat initially warned about requiring Node 22+, while the development machine had Node 20. This caused intermittent compiler crashes and missing function errors. While not fatal, it introduced instability that was difficult to diagnose.

**MetaMask & Network Integration:**
- **Nonce Desync**: MetaMask caches transaction counts (nonces) from previous chain instances. After restarting the Hardhat node, transactions would fail silently because MetaMask submitted stale nonces. This was solved by querying the nonce directly from the Hardhat node's JSON-RPC endpoint, bypassing MetaMask's cache entirely.
- **Chain ID / Network Confusion**: MetaMask displayed warnings like "network name may not match chain ID" when connecting to the local Hardhat node. This occurred because MetaMask's built-in network definitions conflicted with Hardhat's default chainId (31337). Manually configuring a custom network in MetaMask with the correct RPC URL and chain ID resolved the warnings.

**Development Workflow:**
- **Hardcoded Addresses**: Early development required manually copying the deployed contract address and pasting it into the frontend configuration after every redeployment. This was error-prone and slow. It was solved by having the deploy script automatically patch the frontend's `contract.js` file.
- **UI Role Synchronisation**: Ensuring the UI correctly reflected admin vs. voter status in real-time when users switched MetaMask accounts required subscribing to the `accountsChanged` event and re-fetching all contract state on every switch.
- **Test Isolation**: Early test attempts had state leaking between test cases (e.g. a candidate added in one test affecting another). This was solved by using Hardhat's `loadFixture` pattern, which snapshots and restores chain state between tests.

### Key Learnings
- **Environment Stability Matters**: The biggest time sink was not writing code — it was getting the toolchain to work. Dependency conflicts between Hardhat versions, plugin ecosystems, and Node.js versions consumed significant development time. The lesson: always verify version compatibility before starting, and pin dependency versions in `package.json`.
- **Web3 Identity**: In blockchain applications, a "login" is fundamentally different from traditional systems. There is no username or password — identity is a cryptographic key pair, and authentication is a signed transaction.
- **State Machine Design**: Managing the election lifecycle (Not Started → Active → Ended) across both the smart contract and the frontend required careful state synchronisation and defensive checks at every boundary.
- **Infrastructure Automation**: Robust deployment and initialisation scripts are as important as the smart contract itself. Automating address updates and voter registration eliminated an entire class of manual errors.
- **Immutability Trade-offs**: Once a smart contract is deployed, its code cannot be changed. This forced us to think carefully about edge cases during design, since bugs cannot be patched post-deployment.

### Future Implications
- **Zero-Knowledge Proofs**: Future versions could use ZK-SNARKs to enable private voting — where the blockchain verifies that a vote is valid without revealing which candidate was chosen.
- **Decentralised Governance (DAO)**: The system could be extended so that voters themselves propose candidates or modify election rules through on-chain governance proposals.
- **Mainnet Deployment**: Moving from a local Hardhat node to an Ethereum testnet (Sepolia) or Layer 2 (Arbitrum, Optimism) would allow real-world testing with actual network conditions and gas costs.
